My Favourite Language Features

When someone asks me what my favourite programming is, it is really difficult to answer them because every language has its pros and cons. I have never really like a specific programming language as a whole, but rather I like certain parts of a language. For instance, I really like how Java is cross-platform. However, the fact that it does not have lambda expressions and operator overloading makes me sad. The following is a list of special language features that I really like, and I think more programming languages should adopt these features.

Properties

As fellow programmers, we all know that an object should protect its members from modifications outside of the class. As a result, members are set to private so that they are only accessible within the class. To allow others to read or write private members of the object, getters and setters are introduced. However, this leads to the overhaul of creating methods that are really annoying to write and to use. Consider the following class written in c++.

class A {
  int val;
public:
  int getVal() {
    return val;
  }
  void setVal(int val) {
    this.val = val;
  }
}

The code above does not seem too terrible. However, imagine adding 10 more members. Besides declaring the 10 variables (which are obviously inevitable), we also need to declare 20 extra methods. At the end, you will have hundred lines of garbage sitting at the bottom of the class. Languages with propertes provide a convenient way to get/set members. It is a quick way to define members along with their getters and setters. Consider the following class written in C#.

1
2
3
class A {
  public int val { get; private set; }
}

The above code defines the property val with public read and private write access. Unlike getter/setter methods, properties can be treated like variables. For instance, to add val by 1, you can simply write val++. However, to add val in c++, you will have to write setVal(getVal() + 1), which is a lot more annoying to write.

Generators

After a program leaves a method, the local stack is lost, meaning that all local variables have been lost. When the program re-enters the method, it must start over from the beginning. Generators allow the program to continue from where the program last temporary leaves the method. If you are a Python or Ruby user, you should be no stranger to generators. Consider the following code written in Python.

1
2
3
4
5
6
7
def generate(num):
  while num > 0:
    yield num
    num -= 1

for i in generate(1000000):
  print i

The above code print from 1000000 to 1. Note that the program temporary exits generate upon yield and continues execution of generate when the for loop enters generate again. Without generators, one would have to either return an array with 1000000 numbers or to turn generate into an object and store state variables within the object.

Keyword Arguments

When I first learned Python, I thought keyword arguments was the best design ever. It solves a lot of code readability issues of functions with lots of arguments. Imagine a class with a constructor that takes in 10 optional arguments. The arguments become extremely difficult to keep track of. Consider the following example written in c++.

1
2
3
4
5
6
7
8
void doStuff(string* a, string* b, string* c, string* d, string* e, string* f, string* g, string* h, string* i, string* j) {
  // do stuff
}

string x = "x";
string y = "y";
string z = "z";
doStuff(&x, NULL, NULL, NULL, &y, NULL, NULL, NULL, NULL, &z);

Are you able to figure out which arguments x, y, and z are assigned to? You are probably more inclined to find the person who wrote this code and punch the person in the face. Writing out all the NULLs would also drive people nuts. Now, you might find this stupid and ask, "why not overload the function to only accept three arguments?" Overloading the function definitely helps, but it cannot always save the world. What if I want to pass in a, b, and c, but occasionally I want to pass in d, e, and f instead? Function overload would not work because the type of the arguments is the same. The program will not compile due to ambiguity. Now, it is time for keyword arguments to the rescue. The above code can be rewritten into the following in Python.

1
2
3
4
5
6
7
def doStuff(a=None, b=None, c=None, d=None, e=None, f=None, g=None, h=None, i=None, j=None):
  # doStuff

x = 'x'
y = 'y'
z = 'z'
doStuff(a=x, e=y, j=z)

Now, we can clearly see that x, y, and z are assigned to a, e, and j respectively. The caller can also rearrange the order of the arguments arbitrarily, i.e. doStuff(j=z, a=x, e=y), because all arguments are unambiguous regardless of the order and the number of arguments.

My Rant About JavaScript

When people say JavaScript is the language of the future, it really scares me. It REALLY does. I definitely do not have the qualifications to claim that JavaScript is crap because I do not have 10 years of JavaScript experience. However, I have written enough JavaScript to tell you that I am not a big fan of JavaScript. Nevertheless, one thing is for sure. Whenever someone touches web development, JavaScript is inevitable. If you are curious enough to look at how I created this website, that's right. I used AngularJS. How is it possible to make my entire website into a single page web application? JavaScript!

The following is a list of reasons why JavaScript is not one of my favourite programming languages.

The Curse of Internet Explorer (IE) 7/8

That's right. Every web developer hates the guts of IE 7 and 8. A web developer would spend 30% of their time to develop new features while spending 70% of their time trying to get the features working in IE 7 and 8. If you ever have to support IE 6, please write down the suicide hotline number on your desk. Anyway, you would think that the plague only applies to CSS, but no! IE 7 and 8 also manage to plague JavaScript developers. So, why is IE 7 and 8 so bad? Does Microsoft always make terrible browsers? No! IE 10 is not bad at all. However, has every user migrated to IE 10? No, so this is where the problem lies. Microsoft never forced its users to upgrade their browsers. As a result, new features and bug fixes cannot be applied, and IE 7 and 8 became a piece of crap.

Did you know that JavaScript have foreach loops and generators? That's right! They are new language features adopted by all modern browsers. So, we can use them, can't we? Your boss would smile at you and say, "nope, we need to support IE 7 and 8." Then, you will have to download JQuery and use the each function. So, I heard you like generators. No! You can't use it at all! Not even JQuery can save you this time. At the end of the day, we all have to stick with the ECMA 3rd edition standard created since 2000.

Object-Oriented Programming (OOP) in JavaScript

If you ever write a web application and think that you do not need good code design because the code is at most 100 lines long, think again. A year after, if you are still on this project, it will come back and bite you. I have seen a web application that has grown into 2500 lines of spaghetti because the original developer thought that the code is going to be short. Guess who is responsible for converting the spaghetti into properly designed code. Me.

I am sure that every programmer knows the benefits of OOP. Reusability. Modularity. If you want to look for something in a cluster of spaghetti, you have to dig through the entire code. If everything is separated into classes, you know exactly where to start searching. If you are new to JavaScript, you would think that JavaScript does not have objects, but it does. Every function is an object. Personally, I find it very bizarre. Take a look at the following JavaScript code.

var A = function() {
  this.val = 0;

A.prototype.add = function() {

The code above defines class A with member function add. If we only look at line 1 to 3, who would have thought that A is a class? In my opinion, there should be a clear distinction between a class and a function. A class is a class. A function is a function. What benefits do you get from mixing the two? I will never use A as a standalone function. Whenever I write a class in JavaScript, I will always have to add the comment // class Blah above the class definition to provide distinction between a class and a function.

Access Modifiers

Are there any access modifiers in JavaScript? The answer is "sort of". Information hiding can be done in two ways. The first way is to use underscores like how everybody does it in Python. Let's use the same example as above and add some information hiding.

 1
 2
 3

10
var A = function() {
  this._val = 0;
};

A.prototype.add = function() {
  this._val++;
};

var a = new A();
a.add();

I am already annoyed by the fact that whenever I want to access member variables, I have to specify this. Now, I have to add an extra underscore to every member variable access. Some people like adding underscores to denote private member variables even if the language provides access modifiers. I personally find it unncessary because how often would you have public member variables? If you do, your code would become very inconsistent because you would use getters and setters for private member variables and direct access for public member variables.

The second way of information hiding is to declare local variables within the constructor. Wait, what? That's right! In c++, this is complete garbage. However, in JavaScript, the following code actually works.

1
2
3
4
5
6
7
8
9
var A = function() {
  var val = 0;
  this.add = function() {
    val++;
  };
};

var a = new A();
a.add();

With this code, val is only accessible by add(). However, note that if add() is declared as a prototype function like the example above, val will not be available to the prototype function. So, we have private members that are not available to the class itself. Isn't this "awesome"? If that is the case, why would one ever use prototype functions? add() from the sample code above is public anyway. In fact, we use prototypes for better performance, and perhaps inheritance. Note that prototypes are created once only, whereas functions within the constructor are created every time when a new instance is created.

To sum up, information hiding is possible in JavaScript. However, it is very hacky and developers are required to put much more efforts than necessary into information hiding.
